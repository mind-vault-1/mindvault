# @mindvault/registry-client

Typed TypeScript client for the MindVault [`vault-registry`](../../contract/contracts/vault-registry) Soroban contract.

This package wraps the bindings generated by `stellar contract bindings typescript`. It exists so that `server/`, `web/`, and `mcp/` can depend on a single stable import (`@mindvault/registry-client`) instead of reaching into a `generated/` directory.

## Structure

```
packages/registry-client/
├── src/
│   ├── index.ts          # thin wrapper + factory; re-exports everything from generated/
│   └── generated/        # output of `pnpm contract:bindings` (commit to git)
├── package.json
├── tsconfig.json
└── README.md
```

## Usage

```ts
import { createRegistryClient } from "@mindvault/registry-client";

const registry = createRegistryClient({
  contractId: process.env.VAULT_REGISTRY_CONTRACT_ID!,
  rpcUrl: process.env.SOROBAN_RPC_URL!,
});

const resource = await registry.get({ id: "abc123" });
console.log(resource.result);
```

For write calls (`register`, `set_price`, `update_metadata`, `transfer_ownership`) you'll need to provide a signer — see the generated `Client` docs and the `@stellar/stellar-sdk` transaction-signing examples.

## Regenerating bindings

Bindings are produced from the **deployed** contract on testnet. After a new deploy, run from the repo root:

```bash
pnpm contract:bindings
```

This writes fresh code into `src/generated/`. Commit the result so reviewers see the diff.

## Why a separate package?

- **Single source of truth** for the on-chain interface — every consumer pins the same generated types.
- **Decoupling** — replacing the bindings or upgrading the contract doesn't ripple through every consumer's import paths.
- **Future-proofing** — additional contracts (escrow, categories, etc.) can each get a sibling package under `packages/` without bloating any single consumer.

## Build

```bash
pnpm --filter @mindvault/registry-client build
```
